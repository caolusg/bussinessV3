# 业务逻辑与数据库规划

更新时间：2026-04-11

## 1. 产品定位

本项目是一个“国际贸易场景下的商务中文自由练习平台”，不是闯关考试系统。

核心目标：

- 学员围绕外贸业务全流程练习中文沟通。
- 学员可以自由选择任意业务环节，不需要按顺序解锁。
- 每个业务环节都可以无限次练习，不设置“完成”“通关”“第几次尝试”的限制。
- 系统提供 AI 客户/采购方作为对话对象。
- 系统提供 AI 教练反馈、词汇、常用句式、外贸常识等学习支持。
- 老师后续可以维护业务场景、教学资源、AI 提示词和学生练习数据。
- 系统需要尽可能完整地记录练习过程、资源使用、AI 交互和反馈结果，未来用于科研数据分析。

## 1.1 科研数据采集目标

本系统未来不仅是练习平台，也是科研数据采集平台。

需要记录的数据不应只限于“最终聊天记录”，还应包括：

- 学员在什么时间进入了哪个业务环节。
- 学员查看了哪些学习资源。
- 学员在发送消息前后经历了哪些交互。
- 学员每轮输入的原始文本。
- AI 对手每轮回复的文本。
- AI 教练给出的结构化反馈。
- AI 调用时使用的模型、provider、prompt 版本和是否降级。
- 学员是否查看了 AI 教练、小组讨论、学习资源。
- 每个阶段练习的时间、轮次、行为路径。
- 学员画像信息，如国籍、HSK 等级、专业、学习背景。

科研分析可能关注的问题：

- 不同 HSK 等级学生在外贸业务表达上的差异。
- 不同国家或母语背景学生的商务中文策略差异。
- 学习资源查看行为是否影响后续表达质量。
- AI 教练反馈是否改善下一轮回复。
- 哪些业务环节最容易出现沟通失败。
- 学员是否倾向于反复练习某些环节。
- 常见错误类型、让步策略、术语使用情况。

因此，数据库设计应遵循：

- 练习端不做通关限制，但数据端要完整记录行为。
- 尽量记录“过程数据”，不要只记录结果。
- AI 相关数据要保留输入、输出、模型、prompt 版本和降级状态。
- 面向分析的数据可以使用 JSONB 保存早期不稳定结构。
- 后续研究指标稳定后，再拆分成更规范的分析表。

## 2. 当前业务流程

当前设计包含 8 个外贸业务环节：

1. 获客：`acquisition`
2. 报价：`quotation`
3. 磋商：`negotiation`
4. 合同：`contract`
5. 备货：`preparation`
6. 报关：`customs`
7. 结算：`settlement`
8. 售后：`after_sales`

当前前端规则：

- 默认进入第 1 环节“获客”。
- 学员可以点击任意环节练习。
- 工作流地图只表示当前选中的环节，不表示完成度。
- 任务卡只展示练习目标和提示，不展示完成状态。
- 侧边栏学习资源包含三类：商务词汇、常用句式、外贸常识。
- 学习资源现在仍在前端常量中，后续需要迁移到数据库。

当前后端规则：

- 每个学员、每个环节会创建或复用一个 active simulation session。
- 学员发送消息后，后端保存学员消息。
- 后端调用 AI 生成对手回复，并保存为 `role='opponent'` 的消息。
- AI 结构化反馈保存到消息字段：
  - `coach_note`
  - `assessment_json`
  - `trace_json`
  - `persona_json`

## 3. 明确不做的业务规则

以下规则当前不应该引入：

- 不做顺序解锁。
- 不做任务通关。
- 不做任务完成状态。
- 不做最大尝试次数。
- 不在学生端显示“第 1/3 次尝试”。
- 不把练习次数作为限制条件。

后续如果要统计练习情况，只做“学习分析”和“练习记录”，不要变成闯关门槛。

## 4. 当前数据库现状

当前有效 Prisma schema：

```text
apps/api/prisma/schema.prisma
```

根目录下的旧 schema：

```text
prisma/schema.prisma
```

根目录 schema 是旧占位文件，后续应删除、归档或明确标注为废弃，避免误用。

当前已有主要数据表：

### 用户与角色

```text
users
roles
user_roles
student_auth
student_profile
```

用途：

- 学生和教师登录。
- 用户角色区分。
- 保存学生基础资料。

### 练习会话与消息

```text
simulation_sessions
simulation_messages
```

用途：

- 保存每个学生在每个业务环节的练习会话。
- 保存学生消息、AI 对手回复和 AI 教练结构化反馈。

当前问题：

- `simulation_sessions.attempt_no` 仍然存在，这是旧的“尝试次数”设计遗留。
- 当前产品已经不需要尝试次数限制。
- 可以暂时保留字段以降低迁移风险，但新业务逻辑不应再依赖它。

### 旧通用聊天

```text
conversations
messages
```

用途：

- 旧的通用聊天表。

当前建议：

- 暂时保留。
- 后续确认没有业务依赖后，再决定删除或改造成通用 AI 对话/教师工具表。

## 5. 目标数据库设计

### 5.1 业务阶段表：`business_stages`

用途：

- 存储 8 个外贸业务环节。
- 让前端工作流地图从数据库读取，而不是写死在前端常量里。

建议字段：

```text
id UUID primary key
key TEXT unique not null
sort_order INT not null
title_zh TEXT not null
title_en TEXT
description TEXT
is_active BOOLEAN default true
created_at TIMESTAMPTZ default now()
updated_at TIMESTAMPTZ
```

示例数据：

```text
key=acquisition, sort_order=1, title_zh=获客, title_en=Client Acquisition
key=quotation, sort_order=2, title_zh=报价, title_en=Quotation
key=negotiation, sort_order=3, title_zh=磋商, title_en=Negotiation
key=contract, sort_order=4, title_zh=合同, title_en=Contract
key=preparation, sort_order=5, title_zh=备货, title_en=Preparation
key=customs, sort_order=6, title_zh=报关, title_en=Customs
key=settlement, sort_order=7, title_zh=结算, title_en=Settlement
key=after_sales, sort_order=8, title_zh=售后, title_en=After-sales
```

注意：

- 不需要 `locked` 字段。
- 不需要 `completed` 字段。
- `is_active` 只表示该阶段是否显示，不表示学生进度。

### 5.2 阶段任务表：`stage_tasks`

用途：

- 存储每个阶段的练习目标、提示和任务卡内容。

建议字段：

```text
id UUID primary key
stage_id UUID references business_stages(id)
task_code TEXT unique not null
title TEXT not null
goal TEXT not null
sub_goal TEXT
tip_title TEXT
tip_content TEXT
is_default BOOLEAN default true
is_active BOOLEAN default true
created_at TIMESTAMPTZ default now()
updated_at TIMESTAMPTZ
```

业务规则：

- 一个阶段可以先只有一个默认任务。
- 后续老师可以为同一阶段配置多个任务场景。
- 学生自由选择阶段时，默认加载 `is_default=true` 的任务。

### 5.3 学习资源表：`learning_resources`

用途：

- 存储商务词汇、常用句式、外贸常识。
- 替代当前前端 `STAGE_RESOURCES` 常量。

建议字段：

```text
id UUID primary key
stage_id UUID references business_stages(id)
type TEXT not null
term TEXT not null
explanation TEXT not null
example TEXT
sort_order INT default 0
is_active BOOLEAN default true
created_at TIMESTAMPTZ default now()
updated_at TIMESTAMPTZ
```

`type` 可选值：

```text
vocabulary
phrases
knowledge
```

业务规则：

- 每个阶段至少应有三类资源。
- 每类资源可以有多条。
- 资源不和学生进度绑定。
- 资源点击只记录学习行为，不限制练习。

### 5.4 AI 场景配置表：`stage_ai_scenarios`

用途：

- 存储每个业务阶段的 AI 对手设定和系统提示词。
- 让教师端后续可以维护 AI 角色和难度。

建议字段：

```text
id UUID primary key
stage_id UUID references business_stages(id)
name TEXT not null
opponent_name TEXT
opponent_role TEXT
system_prompt TEXT not null
difficulty TEXT default 'standard'
is_default BOOLEAN default true
is_active BOOLEAN default true
created_by UUID references users(id)
created_at TIMESTAMPTZ default now()
updated_at TIMESTAMPTZ
```

业务规则：

- 每个阶段先有一个默认 AI 场景。
- 未来教师可以新增不同风格的场景。
- `difficulty` 只影响 AI 回复风格，不影响学生是否能进入练习。

### 5.5 练习会话表调整：`simulation_sessions`

当前已有：

```text
simulation_sessions
```

建议后续调整：

```text
保留：
id
user_id
stage
status
created_at
updated_at

新增可选：
stage_id UUID references business_stages(id)
task_id UUID references stage_tasks(id)
scenario_id UUID references stage_ai_scenarios(id)
title TEXT

弱化或后续删除：
attempt_no
```

业务规则：

- 不再用 `attempt_no` 表达“第几次练习”。
- 学生可以长期复用同一个 active session。
- 如果后续要支持“新开一段练习”，可以新增 archive/restart 功能，但仍不叫“尝试次数”。

### 5.6 练习消息表调整：`simulation_messages`

当前已有：

```text
simulation_messages
```

当前字段基本可继续使用：

```text
id
session_id
role
content
coach_note
assessment_json
trace_json
persona_json
turn_index
created_at
```

后续可新增：

```text
message_type TEXT
metadata_json JSONB
```

建议：

- 先不要过度拆分 AI 反馈表。
- 当前 `assessment_json`、`trace_json`、`persona_json` 对迭代速度更友好。
- 等 AI 反馈结构稳定后，再考虑拆成结构化评分表。

### 5.7 学习行为记录表：`practice_events`

用途：

- 记录学生看了哪些资源、进入了哪些阶段、请求了哪些教练反馈。
- 用于教师端分析，不用于限制学生。
- 用于科研分析学生练习路径和资源使用行为。

建议字段：

```text
id UUID primary key
user_id UUID references users(id)
stage_id UUID references business_stages(id)
session_id UUID references simulation_sessions(id)
event_type TEXT not null
resource_id UUID references learning_resources(id)
metadata_json JSONB
created_at TIMESTAMPTZ default now()
```

`event_type` 示例：

```text
stage_opened
message_sent
resource_viewed
coach_viewed
discussion_viewed
simulation_started
simulation_exited
ai_reply_received
```

业务规则：

- 只记录行为。
- 不产生完成状态。
- 不用于解锁。
- 不阻塞用户操作，记录失败不应影响练习。

建议 `metadata_json` 内容示例：

```json
{
  "stage_key": "quotation",
  "resource_type": "phrases",
  "resource_term": "如果订单数量增加，我们可以讨论阶梯价格。",
  "client_time": "2026-04-11T14:00:00.000Z",
  "route": "/dashboard"
}
```

### 5.8 AI 调用记录表：`ai_interaction_logs`

用途：

- 记录每次 AI 调用的关键上下文。
- 支持后续科研复盘、模型效果比较和 prompt 版本分析。

建议字段：

```text
id UUID primary key
user_id UUID references users(id)
session_id UUID references simulation_sessions(id)
message_id UUID references simulation_messages(id)
stage_id UUID references business_stages(id)
provider TEXT
model TEXT
prompt_version TEXT
system_prompt TEXT
input_messages_json JSONB
output_text TEXT
output_json JSONB
latency_ms INT
degraded BOOLEAN default false
error_code TEXT
error_message TEXT
created_at TIMESTAMPTZ default now()
```

说明：

- `system_prompt` 用于保留当时真实使用的 prompt。
- `input_messages_json` 用于保留发送给模型的上下文，但后续要考虑脱敏策略。
- `output_text` 保存模型对手回复。
- `output_json` 保存结构化反馈、评分、风险、优势等内容。
- `degraded=true` 表示使用了 fallback 或 AI 调用失败。

科研价值：

- 比较不同 prompt 版本对学生回复质量的影响。
- 分析 DeepSeek 等模型在不同业务环节的回复稳定性。
- 复盘 AI 教练反馈是否对学生后续表达有帮助。

隐私注意：

- 如果真实学生数据进入研究，需要在导出时做匿名化。
- 不应在日志中保存 API key。
- 可以在导出研究数据时把 `user_id` 替换为研究编号。

### 5.9 消息分析结果表：`message_analysis_results`

用途：

- 保存对每条学生消息的结构化分析结果。
- 避免未来每次科研统计都重新调用 AI 分析。

建议字段：

```text
id UUID primary key
message_id UUID references simulation_messages(id)
user_id UUID references users(id)
session_id UUID references simulation_sessions(id)
stage_id UUID references business_stages(id)
analysis_version TEXT
language_quality_json JSONB
business_strategy_json JSONB
trade_term_usage_json JSONB
error_tags_json JSONB
score_json JSONB
created_at TIMESTAMPTZ default now()
```

可分析维度示例：

```json
{
  "language_quality": {
    "fluency": 4,
    "accuracy": 3,
    "appropriateness": 4
  },
  "business_strategy": {
    "responded_to_client_concern": true,
    "used_concession_exchange": false,
    "clarified_trade_terms": true
  },
  "error_tags": ["价格依据不足", "让步条件不明确"]
}
```

说明：

- 第一阶段可以先不做自动分析表。
- 但表结构应该提前规划，因为科研分析会需要稳定字段。
- 前期可以先把分析结果放在 `simulation_messages.assessment_json` 中，后期再迁移到此表。

### 5.10 学生学习画像表：`student_learning_snapshots`

用途：

- 后续用于 AI 个性化。
- 保存系统对学生能力、薄弱点、常见问题的总结。

建议字段：

```text
id UUID primary key
user_id UUID references users(id)
summary_json JSONB
strengths_json JSONB
weaknesses_json JSONB
stage_preferences_json JSONB
resource_usage_summary_json JSONB
updated_at TIMESTAMPTZ
```

建议：

- 不是第一阶段必须做。
- 等练习数据积累后再做。

### 第 1 步：课程内容落库

新增：

```text

learning_resources
```

同时新增 seed：

```text
apps/api/scripts/seed-content.mjs
```

迁移内容：

- 当前 8 个阶段。

- 当前商务词汇、常用句式、外贸常识。

新增 API：

```text
GET /api/content/stages
GET /api/content/stages/:stageKey/task
GET /api/content/stages/:stageKey/resources
```

前端改造：

- 工作流地图从 API 获取阶段。
- 任务卡从 API 获取任务。
- 资源面板从 API 获取资源。

### 第 2 步：AI 场景配置落库

新增：

```text
stage_ai_scenarios
```

迁移内容：

- 当前 `compatibleAiClient.ts` 中按阶段写死的 AI prompt。
- 当前对手信息，如 David / 采购总监。

后端改造：

- AI provider 根据 stage 读取默认 scenario。
- 如果数据库没有配置，保留本地 fallback prompt。

### 第 3 步：练习会话模型清理

调整：

- 新增 `stage_id`、`task_id`、`scenario_id`。
- 新业务不再返回 `attemptNo` 给前端。
- 保留旧字段 `attempt_no` 一段时间，避免迁移风险。

后续再决定是否删除：

```text
attempt_no
simulation_sessions_user_stage_attempt_unique
```

### 第 4 步：学习行为记录与 AI 调用日志

新增：

```text
practice_events

```

记录：

- 打开阶段。
- 查看资源。
- 发送消息。
- 查看 AI 教练。
- 查看小组讨论。
- AI 请求、AI 回复、AI 失败和 fallback。

注意：

- 只做分析，不做完成状态。

### 第 4.5 步：消息分析结果

新增：

```text
message_analysis_results
```

用途：

- 为科研分析沉淀结构化指标。
- 可以先人工或离线脚本生成，不一定第一版实时生成。

### 第 5 步：教师端配置

基于前面的表扩展教师端：

- 管理阶段任务。
- 管理资源。
- 管理 AI 场景 prompt。
- 查看学生练习记录。

## 7. 不建议现在做的事

暂时不建议：

- 立刻删除旧 `attempt_no` 字段。
- 立刻删除旧 `conversations/messages`。
- 立刻把 AI 反馈拆成很多评分表。
- 立刻做复杂的教师端资源编辑器。
- 做任何“完成/通关/解锁”相关字段。

原因：

- 当前产品仍在快速确定内容结构。
- 先把课程内容和资源落库，收益最大、风险最低。
- AI 反馈结构还在变化，用 JSONB 更适合当前阶段。

## 8. 下一步开发建议

下一步应该执行：

1. 新建 Prisma migration，加入 `business_stages`、`stage_tasks`、`learning_resources`。
2. 新增内容 seed 脚本，把当前前端常量导入数据库。
3. 新增 `/api/content` 路由。
4. 前端改为优先从 `/api/content` 拉取阶段、任务和资源。
5. 保留当前前端常量作为短期 fallback，避免数据库未 seed 时页面空白。
6. 第二批 migration 增加 `practice_events` 和 `ai_interaction_logs`，开始记录科研所需过程数据。
