# 更新记录：研究分析 AI 工作台 M1（自然语言查库）

更新时间：2026-05-05

## 本次目标

实现研究分析工作台 M1：研究者可通过自然语言提问，由大模型生成只读 SQL 查询数据库并返回分析结论。

## 后端新增

- 新增路由：`POST /api/research/ai/query`
- 文件：`apps/api/src/routes/researchAi.ts`
- 挂载入口：`apps/api/src/index.ts`

### 路由能力

1. 校验请求参数（`question`）
2. 权限校验（仅 `admin/teacher`）
3. 调用模型将问题转换为 SQL（仅允许 SELECT）
4. SQL 安全校验：
   - 禁止多语句
   - 禁止 DML/DDL（insert/update/delete/drop 等）
   - 仅允许白名单数据表
5. 执行查询并返回行数据
6. 调用模型对结果生成简要中文洞察

## Token / 模型配置说明

大模型 Token 已有可配置入口：

- 环境变量：
  - `AI_API_KEY`（或 `DEEPSEEK_API_KEY`）
  - `AI_PROVIDER`
  - `AI_BASE_URL`
  - `AI_MODEL`
  - `AI_TIMEOUT_MS`
- 管理后台“系统管理”页面也可在线配置 AI 参数并保存。

此外，本次 M1 新增研究查询相关环境变量：

- `RESEARCH_AI_ALLOWED_TABLES`：允许查询的数据表白名单（逗号分隔）

示例：

```bash
RESEARCH_AI_ALLOWED_TABLES=teaching_groups,teaching_group_members,users,student_profile,practice_events
```

## 使用方式（接口）

```bash
curl -X POST http://localhost:8000/api/research/ai/query \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"question":"最近30天各分组活跃趋势"}'
```

可通过两种方式配置：

## M3（多轮上下文增强）

## 本次更新的核心目标（科研导向）

本轮更新的主要目标不是“多一个聊天入口”，而是把平台升级为 **科研分析助理工作台**：

1. 研究者（teacher/admin）可用自然语言提出问题；
2. 大模型将问题转换为可执行 SQL；
3. 系统在只读安全约束下执行查询并返回结果；
4. 再由大模型给出洞察、图表建议、后续追问，形成多轮研究链路。

换句话说，你的意图“借助大模型，像 Codex 一样直接操作底层数据库做分析”已在当前版本落地为 **受控的自然语言→SQL→结果→洞察流程**（受权限和只读规则保护）。

## 具体如何帮助科研

- **降低门槛**：不需要研究者手写 SQL，即可从业务问题直接拿到结构化数据。
- **提高迭代速度**：追问按钮 + 上下文记忆让“假设→验证→再追问”可连续进行。
- **增强可解释性**：返回 SQL 文本、结果行数、耗时、风险提示，便于论文/报告复核。
- **支持产出物**：CSV 导出、图表建议、轻量图表预览，有助于快速形成研究草稿材料。
- **安全可控**：仅 teacher/admin 可用；仅允许 SELECT；白名单表；危险关键字拦截。

## 是否达到“类似 Codex 操作数据库分析”的效果（现状评估）

### 已达到

- 已实现自然语言直接驱动数据库查询与分析返回；
- 已实现多轮上下文分析与追问推荐；
- 已提供研究常用的结果导出与可视化辅助；
- 已具备基础安全防护与风险提示。

### 尚未完全达到（下一步可增强）

- 目前仍是“受控自动 SQL”，不是开放式任意数据库运维代理；
- 图表为轻量预览，尚非完整 BI 图表引擎；
- 反馈数据目前本地存储，未形成跨用户统一评估看板。

结论：当前版本已经达到“**科研数据分析助手（可用）**”目标；若要达到“**全自动数据库研究代理（高级）**”，建议进入下一阶段（M7）做反馈入库、图表组件化、评估监控与灰度策略。
