# 自动科研 Topic Agent 第一版实施记录

Last updated: 2026-05-27

## 目标

在教师后台“自然语言数据分析”中新增一个主动扫描入口，让系统基于现有数据库语义表 `data_table_descriptions` 和一组固定、只读、低风险统计查询，自动生成可研究的科研 topic。

第一版重点是稳定闭环，不做无限制 autonomous SQL 探索。

## 当前状态

第一版已进入当前 `main` 代码。

实际接口：

```text
POST /api/research/ai/discover-topics
```

前端入口位于教师后台“研究分析工作台”的“扫描科研机会”按钮。

## 第一版已实现范围

- 后端新增接口：`POST /api/research/ai/discover-topics`
- 接口读取 `data_table_descriptions`，理解每张表的业务含义、表关系、科研用途和敏感级别。
- 接口执行固定扫描 SQL，覆盖：
  - 表规模
  - 学生画像分布
  - 教学分组规模
  - 实训会话阶段与状态分布
  - 行为事件类型分布
  - AI 调用类型、降级状态、近 30 天趋势
- 将语义说明和扫描结果交给 AI，生成 1-8 个科研 topic。
- 前端在“研究分析工作台”页面新增“扫描科研机会”按钮。
- 前端展示 topic 卡片，包含：
  - 研究题目
  - 研究问题
  - 涉及数据表
  - 核心变量
  - 推荐方法
  - 可行性评分
  - 样本量依据
  - 潜在局限
  - 下一步 SQL
- Topic 卡片已提供“继续分析这个 topic”按钮，会把 topic 标题和研究问题带入现有自然语言查库链路。

## 安全边界

- 第一版扫描 SQL 全部由代码固定，不允许 AI 自行循环探索数据库。
- 只做聚合统计，不直接拉取学生原始文本。
- 固定扫描默认排除管理员、教师、测试角色、停用账号和明显测试账号。
- topic 生成提示要求不在 `nextSql` 中包含 `password_hash`、`email`、`student_no`、`token_hash` 或原始 `user_id`。
- 下一步 SQL 只作为建议展示；用户点击“继续分析这个 topic”后，再通过现有自然语言分析能力继续执行。

## 后续版本

- 为每个 topic 自动生成并执行第一条验证 SQL。
- 增加 topic 收藏、标记已采用、导出研究设计草案。
- 增加定期扫描，比较新数据是否产生新的研究机会。
- 补充一次端到端验收记录：登录教师后台，点击“扫描科研机会”，确认返回 topic、扫描耗时、下一步 SQL 和继续分析链路。
