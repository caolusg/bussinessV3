# 服务器部署说明

Last updated: 2026-04-12

## 目标

本仓库现在支持一套完整的单机部署方案：

- PostgreSQL 16
- Express API
- React/Vite 前端静态站点
- nginx 反向代理前端与 `/api`

部署产物基于以下文件：

- `docker-compose.prod.yml`
- `.env.production.example`
- `apps/api/Dockerfile.prod`
- `apps/web/Dockerfile.prod`
- `apps/web/nginx.conf`

## 推荐服务器环境

- Linux 服务器
- 已安装 Docker Engine
- 已安装 Docker Compose Plugin
- 至少 2 CPU / 4 GB RAM
- 已开放 HTTP 端口，例如 `80`

如果要对外正式提供服务，建议再加一层：

- 云厂商负载均衡
- 或服务器级 nginx / Caddy / Traefik
- 或证书管理工具，用于 HTTPS

当前仓库内置的是应用级 nginx，默认只处理 HTTP 和 `/api` 反向代理。

## 第一步：准备代码

在服务器上拉取当前固定版本代码：

```bash
git clone <your-repo-url>
cd bussinessV3
git checkout main
```

如果你要固定到明确版本，建议额外打 tag 并在服务器上检出该 tag。

## 第二步：准备生产环境变量

复制模板：

```bash
cp .env.production.example .env.production
```

至少修改这些值：

- `POSTGRES_PASSWORD`
- `JWT_SECRET`
- `DEFAULT_TEACHER_PASSWORD`
- `AI_API_KEY`
- `WEB_PORT`
- `APP_BASE_URL`
- `MAIL_MODE`
- `SMTP_HOST`
- `SMTP_PORT`
- `SMTP_SECURE`
- `SMTP_USER`
- `SMTP_PASS`
- `SMTP_FROM`

建议值说明：

- `POSTGRES_PASSWORD`
  - 必须改成强密码。
- `JWT_SECRET`
  - 必须改成长随机串。
- `DEFAULT_TEACHER_PASSWORD`
  - 首次部署可设置为临时密码，登录后再改。
- `AI_API_KEY`
  - 如果留空，系统会降级为本地 fallback 回复。
- `APP_BASE_URL`
  - 用于生成邮件里的验证链接和重置密码链接。
- `MAIL_MODE`
  - 生产环境请使用 `smtp`。
  - `preview` 只适合本地开发，会把链接打到 API 日志和返回值里。
- `SMTP_FROM`
  - 建议使用带域名的发件人，例如 `BIZ Chinese <no-reply@your-domain.example.com>`。
- `CORS_ORIGINS`
  - 如果前端和 API 都走同一域名的 nginx 代理，可以留空。
  - 只有前后端跨域部署时才需要填写多个 origin，用英文逗号分隔。

## 第三步：启动生产容器

从仓库根目录执行：

```bash
npm run prod:up
```

这一步会：

- 启动 PostgreSQL
- 构建 API 生产镜像
- 构建前端静态站点镜像
- 启动 nginx 前端容器

## 第四步：执行数据库迁移和初始化

首次部署必须执行：

```bash
npm run prod:migrate
npm run prod:seed
npm run prod:seed:content
```

这三步分别负责：

- 应用 Prisma migration
- 初始化角色与默认教师账号
- 初始化 8 个业务阶段、任务、学习资源和默认 AI 场景

## 第五步：检查部署结果

检查容器状态：

```bash
docker compose --env-file .env.production -f docker-compose.prod.yml ps
```

检查网页：

```bash
curl http://127.0.0.1:${WEB_PORT:-80}
```

检查 API：

```bash
curl http://127.0.0.1:${WEB_PORT:-80}/api/health
curl http://127.0.0.1:${WEB_PORT:-80}/api/health/db
```

教师登录页：

```text
http://<server-ip-or-domain>:<WEB_PORT>/login/teacher
```

默认教师账号：

- 用户名：`.env.production` 中的 `DEFAULT_TEACHER_USERNAME`
- 密码：`.env.production` 中的 `DEFAULT_TEACHER_PASSWORD`

## 运维常用命令

查看日志：

```bash
npm run prod:logs
```

停止服务：

```bash
npm run prod:down
```

重新部署：

```bash
npm run prod:down
npm run prod:up
npm run prod:migrate
```

## 数据持久化

PostgreSQL 数据保存在 Docker volume：

```text
pgdata_prod
```

这意味着：

- 重启容器不会丢数据
- `docker compose down` 默认不会删除 volume

但如果你执行了带 volume 删除的命令，例如：

```bash
docker compose down -v
```

数据库会被清空。

## 备份建议

建议在服务器上定期执行 PostgreSQL 备份，例如：

```bash
docker compose --env-file .env.production -f docker-compose.prod.yml exec -T db \
  pg_dump -U "$POSTGRES_USER" "$POSTGRES_DB" > backup.sql
```

恢复时可用：

```bash
cat backup.sql | docker compose --env-file .env.production -f docker-compose.prod.yml exec -T db \
  psql -U "$POSTGRES_USER" "$POSTGRES_DB"
```

## HTTPS 建议

当前仓库自带的 web 容器只负责：

- 提供前端静态资源
- 转发 `/api` 到内部 `api:8000`

正式上线建议：

- 把 `web` 容器挂在公网端口
- 再使用外层反向代理处理 HTTPS 证书
- 或者直接用云厂商 LB / CDN 终止 TLS

## 当前不包含的内容

当前部署方案没有自动做这些事情：

- 自动续签 HTTPS 证书
- 自动数据库备份
- 自动滚动发布
- 多机高可用
- 外部对象存储

这套方案适合作为：

- 单台服务器部署
- 教学/研究环境部署
- 内部测试环境
- 第一版生产化落地

## 推荐固定版本方式

建议把当前已验证版本打一个 git tag，再部署该 tag，而不是直接长期跟随 `main`。

推荐流程：

```bash
git tag -a release-2026-04-12 -m "Deployment baseline"
git push origin release-2026-04-12
```

然后服务器上检出：

```bash
git fetch --tags
git checkout release-2026-04-12
```
