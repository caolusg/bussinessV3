# 首次部署流程

这个流程适合“先把代码推到 GitHub，再在服务器上拉代码并通过网页完成初始化”的场景。

## 1. 本地发布

在本地确认代码可用后，提交并推送到 GitHub：

```bash
git add .
git commit -m "Add web-based setup flow"
git push origin main
```

## 2. 服务器拉取代码

第一次部署时，在服务器上克隆仓库：

```bash
cd /bussiness
git clone https://github.com/<your-org>/<your-repo>.git
cd <your-repo>
git checkout main
```

如果仓库已经存在，以后升级版本时直接拉取：

```bash
cd /bussiness/<your-repo>
git pull origin main
```

## 3. 准备生产环境变量

```bash
cp .env.production.example .env.production
```

至少确认这些值：

- `POSTGRES_PASSWORD`
- `JWT_SECRET`
- `APP_BASE_URL`
- `WEB_PORT`
- `DEFAULT_TEACHER_USERNAME`
- `DEFAULT_TEACHER_PASSWORD`
- `AI_API_KEY`，如果要启用真实 AI

## 4. 启动容器

```bash
npm run prod:up
```

如果是第一次部署，数据库通常还是空的。此时访问站点会进入 `/setup`。

## 5. 网页端完成初始化

在浏览器打开站点后：

- 先填写 AI 配置
- 再点击“开始部署/初始化”
- 等待页面显示迁移、建表、写入默认教师账号、写入内容数据完成

初始化完成后，页面会退出 setup 流程，进入正常登录和业务界面。

## 6. 初始化后验证

你可以检查：

- `/setup/status` 返回 `setupComplete: true`
- 数据库里已经有 `roles`、`users`、`user_roles`、`business_stages`
- 默认教师账号可以正常登录

## 关键点

- 教师默认账号不是写死在前端或配置文件里的，而是 bootstrap 时写入数据库。
- AI 配置会保存在运行时目录的状态文件里，适合网页端修改。
- `runtime/` 目录必须持久化挂载，否则初始化状态和网页配置会丢失。
